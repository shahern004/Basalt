# LACI v1.1 Delta Analysis & Basalt Alignment Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Analyze every delta between Basalt v1.0 and LACI v1.1, assess modular update compatibility, identify blockers, and produce actionable alignment tasks so Basalt remains compatible with upstream LACI updates.

**Architecture:** Basalt inherits LACI's inference/web stack layout. LACI v1.1 introduces Docker-network-based service discovery (replacing host-routing), DRY `db/include.yaml` patterns, per-model vLLM includes, a restructured Onyx deployment, and major version bumps across all services. This plan first catalogs deltas, then classifies them as blockers vs. enhancements, then produces alignment tasks.

**Tech Stack:** Docker Compose, vLLM, LiteLLM, Langfuse, Authentik, Open-WebUI, Onyx, Nginx

**Priority:** This alignment work comes BEFORE B2/B3 (RMF context assembly + LLM integration). Rationale: version upgrades (especially LiteLLM v1.81.15 and vLLM v0.17.1) directly affect B3's structured output (`response_format`) behavior. Aligning first avoids building on stale infrastructure.

**Prerequisite:** Push current basalt-stack-v1.0 baseline to local Forgejo git server (planned 2026-03-25). This LACI alignment will be the **first feature branch and PR** on Forgejo. Do NOT start implementation until Forgejo is set up and the baseline is pushed.

**Decisions Made:**
1. **Model loading:** Adopt LACI's HF cache mount pattern (`/root/.cache/huggingface`). Restructure `D:/BASALT/models/` accordingly.
2. **Onyx deployment:** Create `basalt-stack/web/onyx/` mirroring LACI's structure, using upstream `onyxdotapp` images.

## Blocker Analysis: Can Basalt Still Modularly Update from LACI?

### Current Answer: **Yes, with caveats**

Basalt's divergences are concentrated in **configuration and networking**, not in service logic. The services themselves (vLLM, LiteLLM, Langfuse, Authentik) are stock upstream images — Basalt hasn't forked any container code. This means version upgrades are straightforward *if* the surrounding compose and networking are aligned.

### Blockers to Modular Updates

#### BLOCKER 1: Networking Model Mismatch (Severity: HIGH)

**Problem:** LACI v1.1 connects services via two shared Docker networks (`proxy`, `inference-endpoint`). LiteLLM reaches vLLM at `http://gpt-oss-120b:8000/v1` (Docker DNS). Basalt uses `http://host.docker.internal:8001/v1` (host routing).

**Impact:** Every future LACI compose update that references Docker service names will require Basalt to manually translate URLs to `host.docker.internal:PORT`. This is fragile and error-prone.

**Resolution:** Adopt LACI's Docker network model. Create `proxy` and `inference-endpoint` external networks. Update all compose files to attach to appropriate networks. Update `litellm-config.yaml` to use Docker service names.

**Risk:** Basalt's single-GPU, single-model setup means the vLLM service name differs (`vllm` vs. `gpt-oss-120b`). Need a convention for this.

#### BLOCKER 2: Onyx Deployment Divergence (Severity: HIGH)

**Problem:** LACI v1.1 deploys Onyx from `web/onyx/` with a completely restructured compose: LACI-rebuilt backend image, nginx reverse proxy, include files for db/minio/nginx/vespa, and OIDC auth. Basalt deploys from the upstream `onyx/deployment/docker_compose/docker-compose.yml`.

**Impact:** LACI's Onyx updates won't map to Basalt's deployment path. The LACI team rebuilds the Onyx backend image with their own changes — Basalt cannot use these images without access to LACI's build pipeline.

**Resolution Options:**
- **Option A (recommended):** Create `basalt-stack/web/onyx/` mirroring LACI's structure, but using upstream Onyx images. Adopt the nginx/include pattern. This gives structural compatibility while keeping Basalt independent of LACI's image builds.
- **Option B:** Continue using upstream Onyx compose, but this means Onyx updates from LACI can never be applied modularly.

#### BLOCKER 3: Image Registry Pattern (Severity: MEDIUM)

**Problem:** LACI uses `${IMAGE_REPO_BASE}` prefix on all image references, pointing to their private Artifactory registry. Basalt uses bare Docker Hub references.

**Impact:** When applying LACI compose updates, every image reference needs to be stripped of the registry prefix. This is tedious but not technically blocking.

**Resolution:** Adopt `${IMAGE_REPO_BASE}` pattern in all Basalt `.env.example` files. Set it to empty string (`""`) for Docker Hub, or to a local registry URL for air-gap staging. This makes LACI diffs apply cleanly.

#### BLOCKER 4: `db/include.yaml` Pattern (Severity: MEDIUM)

**Problem:** LACI v1.1 uses `include: - ./db/include.yaml` in langfuse and litellm compose files. Basalt has these services inline.

**Impact:** LACI compose diffs won't apply cleanly because the file structure differs. The services exist but in different files.

**Resolution:** Extract database services from Basalt's langfuse and litellm compose files into `db/include.yaml` files matching LACI's structure.

#### BLOCKER 5: vLLM Per-Model Include Pattern (Severity: MEDIUM)

**Problem:** LACI uses `include: - ./gpt-oss-120b/include.yaml` with per-model directories. Basalt has a single flat compose file.

**Impact:** Multi-model support and LACI vLLM updates require restructuring.

**Resolution:** Create `basalt-stack/inference/vllm/gpt-oss-20b/include.yaml` following LACI's pattern. Update the parent `docker-compose.yaml` to use `include:`.

### Non-Blockers (Config-Level Differences, Easy to Reconcile)

- **`drop_params` divergence** — Basalt deliberately set `false` for structured output. LACI uses `true`. This is a **conscious Basalt override** that should be documented but kept.
- **`gpt-4` alias** — Basalt-specific convenience. Keep it.
- **Health check intervals** — Basalt uses 60s (faster detection). LACI uses 300s. Keep Basalt's value.
- **Authentik hardening** — Basalt is ahead. LACI may adopt Basalt's improvements later.
- **Open-WebUI shared-secret** — Basalt-specific Authentik integration. Keep it.

## Task 2: Restructure vLLM to Per-Model Include Pattern

**Files:**

**Context:** LACI v1.1 uses a per-model directory pattern where each model gets its own `include.yaml` with model-specific command args, GPU counts, and environment variables. The parent `docker-compose.yaml` becomes a one-liner that selects which model to load. This allows easy model swapping and multi-model support. Basalt should adopt this to enable future model additions.

- [ ] **Step 1: Create `gpt-oss-20b/include.yaml`**

Adapt from LACI's `gpt-oss-120b/include.yaml`, but sized for Basalt's single RTX A6000 (1 GPU, 48GB VRAM):

```yaml
---
services:
  gpt-oss-20b:
    image: ${VLLM_IMAGE:?error}:${VLLM_TAG:?error}
    restart: unless-stopped
    environment:
      HF_HUB_OFFLINE: "True"
      HF_HUB_DOWNLOAD_TIMEOUT: 1
      HF_HUB_ETAG_TIMEOUT: 1
      CUDA_VISIBLE_DEVICES: all
      VLLM_NO_USAGE_STATS: "1"
    shm_size: 10gb
    ipc: host
    volumes:
      - ${HF_CACHE:?error}:/root/.cache/huggingface
    command: >
      --host 0.0.0.0
      --gpu-memory-utilization 0.85
      --max-model-len 8192
      --max-num-seqs 64
      --tensor-parallel-size 1
      --model openai/gpt-oss-20b
      --served-model-name gpt-oss-20b
    env_file:
      - ../.env
    ports:
      - ${VLLM_API_PORT:-8001}:8000
    logging:
      driver: json-file
      options:
        max-file: "3"
        max-size: "10m"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://0.0.0.0:8000/health/"]
      interval: 30s
      timeout: 3s
      start_period: 600s
      start_interval: 5s
    deploy:
      replicas: 1
      restart_policy:
        max_attempts: 5
        condition: on-failure
        delay: 10s
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
    networks:
      - inference-endpoint

networks:
  inference-endpoint:
    external: true
```

**Key differences from LACI gpt-oss-120b:**
- `tensor-parallel-size: 1` (Basalt has 1 GPU vs LACI's 4)
- `max-model-len: 8192` (smaller model context)
- `max-num-seqs: 64` (Basalt-specific tuning)
- `start_period: 600s` (Basalt loads from 9p bridge — slower)
- No `VLLM_ATTENTION_BACKEND` or timeout env vars (smaller model doesn't need them)
- Added `VLLM_NO_USAGE_STATS` (air-gap telemetry guard from Basalt)

**Decision (confirmed):** Adopt LACI's HF cache mount pattern. Model files at `D:/BASALT/models/` must be organized as a HuggingFace cache structure (i.e., `hub/models--openai--gpt-oss-20b/snapshots/<hash>/`). This aligns with LACI's `HF_CACHE` env var and enables standard `huggingface-cli download` workflow for future model additions.

```bash

# Wait for healthy, then restore:

```

**LiteLLM** (volume: `postgres_data`):

## Appendix B: Basalt-Specific Overrides (Keep, Do NOT Align to LACI)

These are intentional Basalt divergences that should be documented and preserved across LACI updates:

| Override | Basalt Value | LACI Value | Reason |
|----------|-------------|-----------|--------|
| `drop_params` | `false` | `True` | Preserves `response_format` for structured output (B3 RMF) |
| `gpt-4` alias | Present | Absent | Allows OpenAI-compatible clients without reconfiguration |
| Health check interval | 60s | 300s | Faster failure detection on single-GPU system |
| Authentik Redis password | Required | Not set | Security hardening (Basalt Phase D) |
| Authentik resource limits | Set | Not set | Prevents OOM on constrained system |
| Authentik health checks | Full | Not present | Reliability on single-node deployment |
| Open-WebUI shared secret | `AUTHENTIK_SHARED_SECRET` | Not present | SSO integration (Basalt Phase D) |
| `VLLM_NO_USAGE_STATS` | `1` | Not set | Air-gap telemetry guard |
| `tools/rmf-generator/` | Present | Not present | Basalt-specific RMF doc automation |
| `--max-num-seqs 64` | Set | Not set (LACI uses defaults) | Tuned for single A6000 |
| `--served-model-name` | `gpt-oss-20b` | Not set (uses model path) | Explicit name for clarity |
| Authentik port mapping | `80:9000` / `443:9443` | `9080:9000` / `9443:9443` | Basalt uses standard HTTPS port for browser-facing `*.basalt.local` SSO |
| Authentik default image tag | Explicit in `.env` only | Default `2023.10.6` in compose | Basalt must always set `AUTHENTIK_TAG` in `.env` — never rely on compose default |
